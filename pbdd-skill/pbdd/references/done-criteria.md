# Done Criteria

Use this reference before marking a feature, task, bug, or release as done.

## Done Rule

Do not mark work done because code was edited. Mark work done only when the
project brain contains evidence that the intended engineering state was reached.

## Task Done

A task can be `Done` when:

- The requested work is implemented or explicitly canceled.
- Required checks were run or skipped with a stated reason.
- Completion evidence is recorded.
- Dependent brain entries are updated.

## Feature Done

A feature can be `Done` when:

- The spec has acceptance criteria.
- Tasks are done or explicitly canceled.
- Tests or validation evidence exist.
- Risks are resolved, accepted, or documented.
- The feature brain entry links to artifacts and evidence.

## Bug Resolved

A bug can be `Resolved` when:

- The symptom or cause is recorded.
- A fix or accepted non-fix decision exists.
- Verification evidence exists.
- Regression coverage is added when practical.

## Research Knowledge

Research can become `Knowledge` when:

- The question is stated.
- Evidence or sources are summarized.
- The conclusion is recorded.
- Follow-up tasks or risks are created when needed.

## Release Done

A release can be `Released` when:

- Scope is recorded.
- Validation evidence exists.
- Known risks are documented.
- Release notes exist.

## Final Response Checklist

When ending an iteration, report:

- Event handled
- Brain files changed
- Artifacts changed
- Code/tests changed
- Checks run
- Remaining risks
- Suggested next step
